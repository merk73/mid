# Подключение MIDGAS к Supabase

Это руководство переводит редактор из локального прототипа в общий режим: карточки, связи, обложки, журнал и восстановление будут храниться в Supabase и одинаково отображаться на всех устройствах.

## Короткий порядок подключения

1. В своем нынешнем аккаунте Supabase создайте **отдельный проект MIDGAS**.
2. Настройте Email/Password и адреса GitHub Pages по разделу 2 ниже.
3. Выполните целиком файл `supabase/migrations/20260714000100_midgas_editor.sql` через **SQL Editor → New query → Run**.
4. Сразу после него тем же способом один раз выполните `supabase/migrations/20260714000200_seed_existing_registry.sql`. Он переносит 29 текущих карточек и 31 связь, сохраняя их постоянные номера.
5. Зарегистрируйте на сайте первый аккаунт модератора и подтвердите почту.
6. В SQL Editor назначьте его администратором, заменив адрес:

   ```sql
   update public.editor_members
   set role = 'admin'
   where user_id = (
     select id from auth.users where email = 'ВАША_ПОЧТА@example.com'
   );
   ```

7. В **Connect** скопируйте только Project URL и `sb_publishable_...` и передайте их Codex для подключения интерфейса. Secret key и пароль базы не нужны.
8. После интеграции проверьте сценарии `anon`, `pending`, `editor`, `admin` по таблице в разделе 9.
9. При покупке домена не создавайте новую базу: добавьте новый адрес в Auth Redirect URLs, проверьте вход и затем поменяйте Site URL.

Если аккаунт был создан до применения миграции и строки в `editor_members` нет, один раз выполните перед назначением роли:

```sql
insert into public.editor_members (user_id)
select id from auth.users where email = 'ВАША_ПОЧТА@example.com'
on conflict (user_id) do nothing;
```

## Что понадобится

- аккаунт на [Supabase](https://supabase.com/dashboard);
- доступ владельца к проекту GitHub `merk73/mid`;
- два файла миграций из `supabase/migrations/`: сначала `20260714000100_midgas_editor.sql`, затем `20260714000200_seed_existing_registry.sql`.

В браузерный код MIDGAS можно передавать только **Project URL** и ключ вида `sb_publishable_...`. Никогда не помещайте в сайт, GitHub, переписку или скриншоты `sb_secret_...`, `service_role`, пароль базы данных и строку подключения. Publishable key предназначен для публичных клиентов, а реальные права ограничиваются `GRANT` и RLS. Подробнее: [официальное описание API-ключей](https://supabase.com/docs/guides/getting-started/api-keys).

## 1. Создать проект

1. Откройте [Supabase Dashboard](https://supabase.com/dashboard), нажмите **New project**.
2. Выберите организацию, задайте имя, регион поближе к основной аудитории и надежный пароль базы данных.
3. Дождитесь состояния проекта `Healthy`.
4. Пароль базы сохраните в менеджере паролей; для работы сайта он не нужен.

### Нужна ли отдельная почта, если в Supabase уже подключен другой сайт

Нет. В одном аккаунте Supabase можно держать несколько независимых проектов. Для MIDGAS создайте **новый проект** в существующем аккаунте: у него будут собственные база, Auth, Storage, URL и API-ключи, поэтому он никак не смешается с другим сайтом. Это проще для управления и оплаты.

Отдельная почта имеет смысл только если MIDGAS должен принадлежать другой организации или другому владельцу, либо если вы намеренно хотите полностью разделить доступ и биллинг. Для обычного случая это создаст лишь второй аккаунт, которым придется отдельно управлять. Надежнее оставить один основной аккаунт, включить двухфакторную аутентификацию и добавить доверенного совладельца в организацию Supabase.

## 2. Настроить Email + Password и адреса сайта

1. В **Authentication → Sign In / Providers → Email** включите Email provider и вход по паролю.
2. Оставьте **Allow new users to sign up** включенным: новый пользователь сможет зарегистрироваться, но автоматически получит только статус `pending` и не сможет редактировать данные.
3. Рекомендуется включить **Confirm email** и выключить anonymous sign-ins.
4. В **Authentication → URL Configuration** укажите:

   - **Site URL:** `https://merk73.github.io/mid/`
   - **Redirect URLs:** `https://merk73.github.io/mid/**`
   - для текущей локальной проверки: `http://127.0.0.1:43129/**`
   - при необходимости: `http://localhost:43129/**`

Для production Supabase рекомендует точные redirect URL, а шаблоны удобны для локальных и вложенных маршрутов. После стабилизации маршрута подтверждения можно заменить production-шаблон точным адресом. Официальные инструкции: [Email/Password](https://supabase.com/docs/guides/auth/passwords) и [Redirect URLs](https://supabase.com/docs/guides/auth/redirect-urls).

### Что произойдет после покупки или смены домена

База, карточки, пользователи и фотографии **не пропадут и не перенесутся заново**: они привязаны к проекту Supabase, а не к доменному имени сайта. После покупки домена потребуется только разрешить новый адрес для Auth:

1. Сначала добавьте новый домен в **Authentication → URL Configuration → Redirect URLs**, не удаляя старый адрес GitHub Pages.
2. Проверьте регистрацию, подтверждение почты, вход и восстановление пароля уже с нового домена.
3. После проверки замените **Site URL** на новый основной адрес, например `https://midgas.ru/`.
4. Старый адрес GitHub Pages оставьте в Redirect URLs на время перехода; удалите его, когда он больше не используется.
5. Если позже появятся OAuth-провайдеры, отдельно обновите callback URL в настройках каждого провайдера.

Код сайта продолжит использовать тот же Project URL и тот же publishable key, поэтому повторно создавать проект Supabase не нужно. Желательно подключить собственный домен через HTTPS до переключения Site URL.

## 3. Применить миграцию

1. Откройте в репозитории файл `supabase/migrations/20260714000100_midgas_editor.sql` и скопируйте его целиком.
2. В Supabase откройте **SQL Editor → New query**, вставьте SQL и нажмите **Run** один раз.
3. Создайте ещё один **New query**, целиком вставьте `supabase/migrations/20260714000200_seed_existing_registry.sql` и нажмите **Run** один раз. Seed запускается только на пустой таблице `records` и намеренно остановится, если в ней уже есть карточки.
4. Убедитесь, что оба запроса завершились без ошибок. Не запускайте отдельные фрагменты миграций в случайном порядке.
5. Откройте **Database → Tables** и проверьте: в `records` должно быть 29 строк, в `relationships` — 31 строка.
6. Откройте **Database → Advisors → Security** и устраните все новые предупреждения перед подключением сайта.

Миграция должна применяться сначала к отдельному тестовому проекту. Если ее потребуется изменить, создайте следующую миграцию, а не редактируйте уже примененную на рабочей базе.

## 4. Проверить Data API в модели 2026 года

`GRANT` и RLS решают разные задачи: `GRANT` разрешает роли обратиться к объекту через Data API, а RLS определяет, какие строки эта роль увидит или сможет изменить. В новых конфигурациях Supabase автоматическая выдача прав может быть отключена, поэтому миграция MIDGAS выдает минимальные права явно. Официальная схема: [Securing your API](https://supabase.com/docs/guides/api/securing-your-api).

1. Откройте **Integrations → Data API → Settings**.
2. Убедитесь, что Data API включен и `public` входит в exposed schemas.
3. Разрешите доступ только к таблицам и функциям, которые перечислены в миграции. Не включайте автоматические **Default privileges for new entities**: новая таблица не должна случайно становиться публичной.
4. Не добавляйте широкие `GRANT ALL` для `anon` или `authenticated`. Если API отвечает `42501 permission denied`, сначала проверьте exposure и явные `GRANT` из миграции; не отключайте RLS.

На каждой доступной через API таблице RLS должен быть включен. `anon` читает только опубликованные и не удаленные карточки; обычный `authenticated` пользователь со статусом `pending` не получает права редактора; запись разрешена только одобренным `editor`/`admin`. См. [Row Level Security](https://supabase.com/docs/guides/database/postgres/row-level-security).

## 5. Проверить Storage

После миграции откройте **Storage** и убедитесь, что создан бакет обложек карточек (имя указано в migration). Для публичного реестра чтение обложек может быть публичным, но загрузка, замена и удаление должны оставаться доступными только одобренным редакторам.

Проверьте политики `storage.objects` для операций `SELECT`, `INSERT`, `UPDATE` и `DELETE`. Для замены файла через `upsert` Supabase требует одновременно `SELECT + INSERT + UPDATE`; одной политики загрузки недостаточно. Подробнее: [Storage Access Control](https://supabase.com/docs/guides/storage/security/access-control).

## 6. Получить два публичных параметра

Откройте **Connect** проекта либо **Settings → API Keys** и скопируйте:

```text
Project URL: https://<project-ref>.supabase.co
Publishable key: sb_publishable_...
```

Не копируйте `Secret key` и legacy `service_role`. Если Dashboard пока показывает legacy `anon`, лучше создать/использовать новый publishable key. В статическом сайте publishable key неизбежно виден посетителю, поэтому безопасность должна обеспечиваться RLS, а не сокрытием этого ключа.

## 7. Подключить supabase-js к статическому сайту

Supabase официально поддерживает CDN-версию библиотеки:

```html
<script
  src="https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2.110.2"
  integrity="sha384-sgcvElP/LlNRtORGYFLO/9K/bENcpJK19HgvqslWXog6d8f/Va4TpltGiBlGntCM"
  crossorigin="anonymous"></script>
<script src="supabase-config.js"></script>
```

В MIDGAS уже закреплена точная версия `@supabase/supabase-js` и добавлена SRI-проверка. Project URL и publishable key находятся в `web/supabase-config.js`; секретных ключей в браузерном коде нет. Официальная установка: [supabase-js](https://supabase.com/docs/reference/javascript/installing).

После интеграции локальное хранилище останется только временным кэшем/режимом миграции. Источником истины станут таблицы Supabase и Storage.

## 8. Регистрация и ручное одобрение редакторов

Правильный поток доступа:

1. Пользователь регистрируется по email и паролю и подтверждает почту.
2. В `auth.users` появляется учетная запись, а в таблице редакторов — запись со статусом/ролью `pending`.
3. Пользователь может войти, но RLS запрещает ему создавать, изменять, удалять, восстанавливать карточки и загружать файлы.
4. Владелец проекта проверяет заявку и вручную меняет роль на `editor` или `admin` через подготовленную административную операцию/SQL из migration.
5. После одобрения сайт повторно проверяет роль при возврате на вкладку. При необходимости пользователь может обновить страницу; повторный вход обычно не требуется.

Первого администратора назначайте только из SQL Editor под владельцем проекта. Не храните роль в `user_metadata`: эти данные пользователь может менять сам. Для авторизации используйте защищенную таблицу ролей или `app_metadata`. См. [User Management](https://supabase.com/docs/guides/auth/managing-user-data) и предупреждения в [RLS guide](https://supabase.com/docs/guides/database/postgres/row-level-security#helper-functions).

`editor` работает с карточками, связями, обложками и восстановлением. `admin` дополнительно одобряет/отзывает редакторов. Отзыв доступа должен немедленно блокировать новые операции; для чувствительного проекта после отзыва также завершите активные сессии пользователя.

## 9. Обязательная проверка RLS перед публикацией

Создайте три тестовые учетные записи: `pending`, `editor`, `admin`. Выполните проверки через сам сайт и вкладку Network браузера:

| Проверка | Ожидаемый результат |
|---|---|
| Посетитель без входа читает активные карточки и обложки | разрешено |
| Посетитель читает мягко удаленную карточку или аудит | запрещено/пусто |
| Посетитель создает или изменяет карточку | `401/403` либо пустой результат |
| `pending` создает карточку, связь или загружает файл | запрещено |
| `editor` создает и редактирует карточку, связь и обложку | разрешено |
| `editor` пытается назначить себе `admin` | запрещено |
| `admin` одобряет редактора | разрешено |
| Мягко удаленная карточка исчезает из реестра, но есть в восстановлении редактора | разрешено только редактору/admin |
| Физический `DELETE` карточки из браузера | запрещено |

Отдельно проверьте отрицательные сценарии после истечения сессии и после отзыва роли. Для автоматизации Supabase рекомендует транзакционные тесты RLS/pgTAP: [Testing overview](https://supabase.com/docs/guides/local-development/testing/overview).

## 10. Восстановление, исходная версия и аудит

- «Удалить» в интерфейсе выполняет soft delete: карточка скрывается, но строка и ее постоянный номер сохраняются.
- «Восстановить» снимает признак удаления; номер не переиспользуется и не сдвигается.
- Перед изменением сохраняется версия карточки, поэтому ее можно вернуть к исходному опубликованному состоянию.
- Создание, изменение, удаление, восстановление, сброс версии, изменение связей и обложки должны попадать в неизменяемый журнал аудита вместе с `auth.uid()` и временем.
- Обычный редактор не должен иметь права изменять или удалять аудит и старые версии напрямую.

Для аварийного восстановления базы используйте резервные копии Supabase; журнал приложения не заменяет backup. Общая информация: [Database overview and backups](https://supabase.com/docs/guides/database/overview).

## Что передать Codex для финальной интеграции

После выполнения шагов пришлите только:

```text
SUPABASE_URL=https://<project-ref>.supabase.co
SUPABASE_PUBLISHABLE_KEY=sb_publishable_...
MIGRATION_STATUS=applied
```

Дополнительно сообщите email первого пользователя, которого вы уже зарегистрировали и хотите назначить администратором. Пароль пользователя, пароль базы, secret key и `service_role` присылать не нужно. После этого можно подключить реальные Auth, CRUD, связи, Storage, журнал и восстановление к существующему интерфейсу MIDGAS.
